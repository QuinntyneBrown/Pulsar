Pulsar CLI — Implementation Plan

A pulsar command-line tool that authors, validates, and generates the data-only plugins (manifest + JSON Schemas + examples) that Pulsar consumes — reusing Pulsar.Core so the CLI can never disagree with what the running tool accepts.

Status: Proposed Date: 2026-06-20 Implements the on-ramp for: ADR-0001 Stack: System.CommandLine · Microsoft.Extensions (Host/DI/Logging/Options) · SOLID Packaging: .NET global tool (pulsar)

Context — why build this

Pulsar plugins are now pure data (ADR-0001): a pulsar.plugin.json manifest + JSON Schemas + example files, consumed via a built-in adapter (see backend/samples/Pulsar.SampleMessages/manifest). That design explicitly aims for catalogs that are hand-authorable, diffable, and generatable from contracts the downstream team already owns (AsyncAPI / OpenAPI / JSON Schema).

What's missing is the tooling that makes that promise real and low-friction:

  • authoring a manifest + N schemas + N examples by hand is tedious (quick-start §9);
  • examples can silently drift from their schemas — the one risk ADR-0001 flagged; and
  • the "generate from existing contracts" benefit has no on-ramp.

A pulsar CLI fills this gap. Because it references Pulsar.Core, it reuses the same loader and validator the running tool uses — there is one source of truth for "what is a valid plugin."

Decisions taken

  • Scope = Full: authoring (validate, new, gen-example, list) + publish (one-shot to Redis) + import (AsyncAPI / OpenAPI / JSON-Schema → plugin folder).
  • Packaging = .NET global tool: <PackAsTool>true</PackAsTool>, <ToolCommandName>pulsar</ToolCommandName> — matches the existing mvp / create-code-cli tools.
  • Stack: System.CommandLine, command-per-file, Microsoft.Extensions (Hosting/DI/Logging/Options/Configuration), SOLID.

Definition of done

  • New project backend/src/Pulsar.Cli in Pulsar.sln, building on net8.0 under the existing Directory.Build.props (nullable / implicit-usings / latest).
  • pulsar --help lists all verbs; each verb works against the bundled sample.
  • Authoring verbs need only Pulsar.Core; publish adds Pulsar.Redis; import adds spec-parser packages — and those parser deps live only in the CLI (Core stays dependency-light).
  • CLI tests in backend/tests/Pulsar.Tests (xUnit, hand-written doubles, TestConsole), green alongside the existing suite.
  • validate returns meaningful exit codes for CI.

Architecture

Composition root — Microsoft.Extensions

Program.cs does only wiring:

  1. Host.CreateApplicationBuilder(args) — Configuration (appsettings + env + args), Logging, and DI in one place.
  2. services.Configure<PulsarOptions>(config.GetSection(PulsarOptions.SectionName)) — reuse Core's existing PulsarOptions ("Pulsar" section: RedisConnectionString, PluginPath).
  3. services.AddPulsarCore(); and services.AddPulsarRedis(); — reuse the existing extension methods (all singletons via TryAddSingleton) verbatim.
  4. Register each command — services.AddSingleton<ICliCommand, ValidateCommand>(); … each importer — services.AddSingleton<ISpecImporter, OpenApiImporter>(); … plus PluginWriter.
  5. Configure console logging; a global --verbose option raises the level (default Warning → Information/Debug).
  6. Build the host, resolve IEnumerable<ICliCommand>, add each .Build() to a RootCommand, then await rootCommand.InvokeAsync(args).
DI bridge: handlers read parsed values from System.CommandLine and call methods on the DI-resolved command instance (constructor-injected Core services) — no reflection magic. System.CommandLine.Hosting's UseHost is a viable alternative but adds another prerelease surface; the manual wiring above is simpler and fully testable.

SOLID mapping

  • SRP — one verb per file; a command only parses → delegates → renders. Domain logic stays in Pulsar.Core; scaffolding I/O in PluginWriter; formatting in ConsoleReport.
  • OCP — adding a verb = one new ICliCommand file + one DI line. Program.cs iterates IEnumerable<ICliCommand>, so the root assembly never changes. Same for import formats via IEnumerable<ISpecImporter>.
  • LSP — every verb is substitutable as ICliCommand; every importer as ISpecImporter.
  • ISP — ICliCommand is a single method; commands depend only on the few Core services they use, not a god-service.
  • DIP — commands depend on Core abstractions (CatalogLoader, MessagePublishService, IMessageTransport, ISpecImporter) injected via ctor; output via System.CommandLine's IConsole (swappable with TestConsole in tests). Concretes wired only in Program.cs.

Commands

Each verb is thin — new code mostly parses input and delegates to existing Core.

VerbSignatureReusesExit codes
validate pulsar validate <manifest|folder> [--strict] [--warn-only] CatalogLoader.Load; re-check each entry's example with SchemaValidator.Validate / MessageTemplateService.Validate 0 ok · 1 hard load error · 2 advisory mismatch (--strict→1, --warn-only→0)
list pulsar list <manifest|folder> CatalogLoader.Load, LoadedCatalog.Messages, CatalogEntry 0 / 1
gen-example pulsar gen-example <schema.json> [--pointer #/$defs/X] [-o file] CompiledSchema.ParseWithPointer; new SchemaExample.FromSchema (see Core change) 0 / 1
new pulsar new <name> [-o dir] [--adapter json-passthrough|json-envelope] [--message KEY ...] PluginManifest/MessageManifestEntry, BuiltInAdapters.Names, MessageTemplateService.JsonOptions; writes via PluginWriter; self-validates 0 / 1
publish pulsar publish <manifest> <key> [--channel c] [--payload file|--stdin] [--redis conn] PluginManagementService.LoadAsync + PluginHost, IMessageTransport.ConnectAsync, MessagePublishService.PublishAsync → PublishResult 0 / 1; warns (never blocks) on schema mismatch — matches the fault-injection philosophy
import pulsar import asyncapi|openapi|jsonschema <spec> -o dir [--adapter ...] ISpecImporter → PluginWriter; then runs the validate path as a self-check 0 / 1
Errors: a System.CommandLine middleware catches PulsarException (and its subtypes in PulsarExceptions.cs) and writes a friendly message to stderr with exit 1; unexpected exceptions get a generic message + exit 1.

Import design

  • ISpecImporter → { string Format; PluginDraft Import(Stream spec, ImportOptions opts); }, where PluginDraft = { PluginManifest manifest, IReadOnlyList<(string relPath, string json)> schemas, examples }.
  • JsonSchemaImporter — the schema is already JSON Schema: emit one message, copy the schema, derive a channel from its title. Zero new deps.
  • AsyncApiImporter — parse YAML/JSON, walk channels + components.messages.*.payload (payloads are JSON Schema); one message per channel/operation.
  • OpenApiImporter — load with Microsoft.OpenApi.Readers; emit one message per component schema; channel from schema name.
  • PluginWriter is shared by new and every importer (DRY): creates dir/pulsar.plugin.json, dir/schemas/*.schema.json, dir/examples/*.json, then runs validation on the result.

One required Core change (small, justified)

The schema→example logic (BuildSkeleton + the examples/default fallbacks) is currently private inside ManifestPluginLoader.ResolveExample (Plugins/ManifestPluginLoader.cs:200-260). Extract the schema-only part into a public helper so the loader and the CLI share one source of truth:

  • Add backend/src/Pulsar.Core/Messages/SchemaExample.cs → public static string FromSchema(CompiledSchema schema) returning pretty JSON (schema examples[0] → default → property-default/type skeleton → "{}").
  • Refactor ManifestPluginLoader.ResolveExample to keep "explicit example file wins", then delegate to SchemaExample.FromSchema. Net behaviour unchanged; covered by existing tests.

This is the only edit outside the new project — and it removes duplication rather than adding it.

Phasing

  1. P0 — skeleton & seam: project + Pulsar.sln entry + Program.cs host/DI + ICliCommand + RootCommand assembly + exception→exit middleware + list (simplest) + first TestConsole test. Proves the architecture end-to-end.
  2. P1 — validate: highest value, no external deps; wire CI exit codes.
  3. P2 — SchemaExample Core extraction + gen-example + new + PluginWriter.
  4. P3 — import: ISpecImporter + 3 importers, reusing PluginWriter.
  5. P4 — publish (Redis) and packaging (PackAsTool, README "CLI" section).

Testing

  • Add CLI tests to backend/tests/Pulsar.Tests (add a ProjectReference to Pulsar.Cli), matching the repo's single-test-project, xUnit, hand-written-double convention.
  • Drive each command through a RootCommand with System.CommandLine TestConsole; assert exit code + captured output. Use temp dirs for new/import outputs.
  • Fixtures: reuse the bundled sample at backend/samples/Pulsar.SampleMessages/manifest (already copied into test output by the test csproj).
  • Key cases: validate clean→0, drifted example→2, broken manifest→1; new→folder that validate then passes; gen-example honours --pointer; publish mismatch warns but still publishes (assert against a fake transport, mirroring MessagePublishServiceTests).

Verification (manual, end-to-end)

From backend/ (so global.json pins SDK 8.0):

dotnet build
dotnet run --project src/Pulsar.Cli -- validate samples/Pulsar.SampleMessages/manifest/pulsar.plugin.json   # exit 0
dotnet run --project src/Pulsar.Cli -- list     samples/Pulsar.SampleMessages/manifest/pulsar.plugin.json
dotnet run --project src/Pulsar.Cli -- new MyPlugin --message Foo --adapter json-envelope -o ./tmp/myplugin
dotnet run --project src/Pulsar.Cli -- gen-example samples/Pulsar.SampleMessages/manifest/schemas/heartbeat.schema.json
dotnet run --project src/Pulsar.Cli -- import openapi path/to/api.json -o ./tmp/from-openapi
# Redis up (docker compose up -d):
dotnet run --project src/Pulsar.Cli -- publish samples/Pulsar.SampleMessages/manifest/pulsar.plugin.json HeartbeatTelemetry --redis localhost:6379
dotnet test

Confirm the publish on the wire with docker exec -it pulsar-redis redis-cli psubscribe "telemetry.*".

Caveat (machine-specific): Windows Smart App Control in enforce mode can block freshly-built unsigned binaries from dotnet run/test. The CLI is a normal app (not a plugin DLL), but the same SAC behaviour may apply on this machine — it is not a design issue.
